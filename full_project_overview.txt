---------------------------------------------------------
1. PROJECT PURPOSE
---------------------------------------------------------

This is a full-stack Bike Rental System built with:

- FastAPI backend
- React frontend
- PostgreSQL database
- JWT authentication

Main intended use:

- Users can register and log in
- Users can browse bikes
- Users can search and filter bikes
- Users can book bikes for selected dates
- Users can see their own bookings
- Users can cancel their own bookings
- Admins can manage bike inventory
- Admins can view dashboard statistics

Important correction:
- The dashboard is not for all logged-in users.
- The dashboard is admin-only in both backend and frontend.

5. BACKEND FLOW
---------------------------------------------------------

Typical request flow:

1. React sends request through Axios
2. FastAPI route receives request
3. Pydantic validates request body/query data
4. JWT dependency checks protected routes when required
5. SQLAlchemy queries PostgreSQL
6. FastAPI returns JSON response
7. React updates UI

Booking flow in current code:

1. User logs in
2. JWT is stored in browser localStorage
3. User opens booking modal on home page
4. Frontend calculates total price and sends:
   - `bike_id`
   - `booking_date`
   - `return_date`
   - `total_price`
5. Backend gets current user from token
6. Backend checks bike exists
7. Backend checks bike availability equals `available`
8. Backend checks for overlapping `booked` rows for same bike
9. Backend creates booking row
10. Backend changes bike availability to `not available`
11. Frontend refreshes the available bike list

10. CURRENT BACKEND ROUTES
---------------------------------------------------------

Authentication routes:

`POST /register`
- Registers user
- Rejects duplicate email
- Hashes password

`POST /login`
- Verifies email/password
- Returns JWT access token

`GET /current-user`
- Returns token subject email

`GET /me`
- Returns current user details:
  - `id`
  - `name`
  - `email`
  - `role`


Bike routes:

`POST /bikes`
- Admin only
- Creates bike

`GET /bikes`
- Returns paginated bike list
- Query params:
  - `page`
  - `limit`

Response shape:
- `items`
- `total`
- `page`
- `limit`
- `pages`

`PUT /bikes/{bike_id}`
- Admin only
- Updates bike

`DELETE /bikes/{bike_id}`
- Admin only
- Deletes bike

`POST /upload-bike-image`
- Admin only
- Uploads image to Cloudinary
- Returns `image_url`

`GET /search-bike`
- Searches by `bike_name`
- Returns matching bikes

`GET /available-bikes`
- Returns paginated available bikes only

`GET /filter-brand`
- Returns bikes matching `brand`

`GET /filter-price`
- Returns bikes within price range


Booking routes:

`POST /book-bike`
- Logged-in users only
- Uses authenticated user from token
- Checks bike exists
- Checks current bike availability
- Checks overlapping booking date range
- Creates booking
- Marks bike as `not available`

`GET /my-bookings`
- Logged-in users only
- Returns current user's bookings

`DELETE /cancel-booking/{booking_id}`
- Logged-in users only
- User can cancel only own booking
- Sets booking status to `cancelled`
- Sets bike availability back to `available`


Dashboard route:

`GET /admin/dashboard`
- Admin only
- Returns:
  - total users
  - total bikes
  - total bookings
  - available bikes
  - total revenue from bookings with status `booked`

14. REAL CURRENT PROBLEMS / BUGS
---------------------------------------------------------

These are actual code-level issues visible now.

1. `MyBookingsPage.js` is not aligned with paginated `/bikes`

- Frontend expects `GET /bikes` to return an array
- Backend now returns paginated object
- Bike name lookup for bookings can fail

2. Alembic migration chain has duplicate bike image column creation

- `0001_initial_schema.py` already creates `bike_image`
- `0002_add_bike_image_column.py` adds `bike_image` again
- Running migrations from a clean database can fail on revision `0002`

3. Booking availability logic is inconsistent with date-range booking

- Backend checks overlap by dates
- But it also marks whole bike as `not available` after one booking
- That means a future non-overlapping booking may still be blocked because
  bike availability is global, not per date range

4. Cancellation logic can incorrectly reopen bike availability

- Cancelling one booking sets bike availability to `available`
- This ignores whether another active booking exists for the same bike

5. Booking total price is trusted from frontend

- Frontend calculates total price
- Backend stores it without recalculating
- A client could manipulate total price before sending request

6. Search/filter API behavior is inconsistent

- `/bikes` and `/available-bikes` are paginated
- `/search-bike`, `/filter-brand`, and `/filter-price` are not paginated
- This creates inconsistent frontend behavior and API design

7. Cloudinary setup is incomplete by default

- `.env` contains placeholder Cloudinary values
- Image upload endpoint exists
- But upload will fail until real Cloudinary credentials are added

8. UI text encoding/polish issue appears in source strings

- Some frontend files contain mojibake-like text such as broken symbols
- This suggests encoding cleanup may still be needed in UI copy
